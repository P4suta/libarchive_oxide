## [0.2.0](https://github.com/P4suta/libarchive_oxide/compare/v0.1.0...v0.2.0) - 2026-07-19

### Added

- [**breaking**] rebuild v0.2 archive architecture ([#15](https://github.com/P4suta/libarchive_oxide/pull/15))

### Fixed

- enforce extractor resource limits ([#17](https://github.com/P4suta/libarchive_oxide/pull/17))

### Other

- repair 10 GiB streaming soak ([#16](https://github.com/P4suta/libarchive_oxide/pull/16))
- make project documentation concise ([#13](https://github.com/P4suta/libarchive_oxide/pull/13))
- complete post-release hardening ([#11](https://github.com/P4suta/libarchive_oxide/pull/11))

### Changed

- Replaced the v0.1 slice, sink, and buffering APIs with bounded
  `ArchiveDecoder`, `ArchiveEncoder`, `Codec`, and `Pipeline` state machines.
- Added opaque sequential/seek, futures-io, and Tokio adapters, explicit
  spooling, consuming `finish`, and archive-level metadata setters.
- Expanded typed metadata and lossless raw-extension preservation across tar,
  cpio, ar, ZIP, 7z, and ISO 9660.
- Added finite resource limits, strict trailing-data validation, secure
  capability-based extraction, atomic commit, and zeroizing AES secrets.
- Added all cpio header dialects and hardlink normalization, streaming ZIP
  descriptors, solid 7z processing, Joliet/Rock Ridge, PAX/GNU sparse, ZIP64,
  and WinZip AES.
- Moved the workspace to Rust 2024. Core remains `no_std + alloc`, dependency
  free, and Rust 1.85-compatible; std and CLI crates use Rust 1.87.
- v0.2 intentionally removes v0.1 source compatibility and provides no legacy
  feature or compatibility shim.

## [0.1.0] — 2026-07-19

### Added

- Streaming read and write support for tar, cpio, ar, ISO 9660, zip, and 7z.
- gzip, zstd, xz, LZMA2, and LZ4 filters.
- `no_std` core crate.
- High-level extraction and archive creation APIs.
- `oxtar`, `oxcpio`, `oxcat`, and `oxunzip`.
