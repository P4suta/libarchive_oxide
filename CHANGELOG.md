# Changelog

Notable changes are generated by [release-plz](https://release-plz.dev).

## [Unreleased]

### Added

- Added compile-time `FilesystemAdapter` application, explicit platform
  capabilities, typed applied/unsupported/refused/partial/OS-error findings,
  and a Linux `cap-std` reference adapter for mode/time, numeric ownership,
  xattrs, POSIX ACLs, sparse files, links, special files, and atomic commit.
- Added statically composed `FormatProvider` and `CodecProvider` chains, typed
  capability queries, provider-aware
  `Pipeline`/reader/engine/create paths, and bounded downstream conformance tests.
- Added mutually exclusive `portable-codecs` (default) and explicit
  `native-codecs` profiles for gzip, bzip2, zstd, xz/LZMA2, and LZ4, with
  shared conformance/fuzz gates, dependency-graph policy, and benchmark/RSS
  evidence.
- Added bzip2 outer-filter detection, streaming read/write, async/Tokio
  adapters, caller-driven pipeline support, and `oxtar -j`/`--bzip2`.
- Migrated zstd sync, caller-driven Pipeline, futures-io, and Tokio paths to
  the Pure-Rust `ruzstd` backend with a native-dependency CI gate.
- Migrated xz/LZMA2 sync, caller-driven Pipeline, futures-io, and Tokio paths
  to a bounded Pure-Rust `lzma-rust2` codec with allocation-before-use limits,
  deterministic writes, and a native-dependency CI gate.
- Migrated LZ4 frame sync, caller-driven Pipeline, futures-io, and Tokio paths
  to the Pure-Rust `lz4_flex` backend with bounded framing and a
  native-dependency CI gate.

### Fixed

- Made malformed zstd decode errors terminal so mutated frames return a typed
  error instead of reusing decoder state and aborting under libFuzzer.

## [0.2.0](https://github.com/P4suta/libarchive_oxide/compare/v0.1.0...v0.2.0) - 2026-07-19

### Added

- Added compile-time `FilesystemAdapter` application, explicit platform
  capabilities, typed applied/unsupported/refused/partial/OS-error findings,
  and a Linux `cap-std` reference adapter for mode/time, numeric ownership,
  xattrs, POSIX ACLs, sparse files, links, special files, and atomic commit.- [**breaking**] rebuild v0.2 archive architecture ([#15](https://github.com/P4suta/libarchive_oxide/pull/15))

### Fixed

- enforce extractor resource limits ([#17](https://github.com/P4suta/libarchive_oxide/pull/17))

### Other

- repair 10 GiB streaming soak ([#16](https://github.com/P4suta/libarchive_oxide/pull/16))
- make project documentation concise ([#13](https://github.com/P4suta/libarchive_oxide/pull/13))
- complete post-release hardening ([#11](https://github.com/P4suta/libarchive_oxide/pull/11))

### Changed

- Replaced the v0.1 slice, sink, and buffering APIs with bounded
  `ArchiveDecoder`, `ArchiveEncoder`, `Codec`, and `Pipeline` state machines.
- Added opaque sequential/seek, futures-io, and Tokio adapters, explicit
  spooling, consuming `finish`, and archive-level metadata setters.
- Expanded typed metadata and lossless raw-extension preservation across tar,
  cpio, ar, ZIP, 7z, and ISO 9660.
- Added finite resource limits, strict trailing-data validation, secure
  capability-based extraction, atomic commit, and zeroizing AES secrets.
- Added all cpio header dialects and hardlink normalization, streaming ZIP
  descriptors, solid 7z processing, Joliet/Rock Ridge, PAX/GNU sparse, ZIP64,
  and WinZip AES.
- Moved the workspace to Rust 2024. Core remains `no_std + alloc`, dependency
  free, and Rust 1.85-compatible; std and CLI crates use Rust 1.87.
- v0.2 intentionally removes v0.1 source compatibility and provides no legacy
  feature or compatibility shim.

## [0.1.0] — 2026-07-19

### Added

- Added compile-time `FilesystemAdapter` application, explicit platform
  capabilities, typed applied/unsupported/refused/partial/OS-error findings,
  and a Linux `cap-std` reference adapter for mode/time, numeric ownership,
  xattrs, POSIX ACLs, sparse files, links, special files, and atomic commit.- Streaming read and write support for tar, cpio, ar, ISO 9660, zip, and 7z.
- gzip, zstd, xz, LZMA2, and LZ4 filters.
- `no_std` core crate.
- High-level extraction and archive creation APIs.
- `oxtar`, `oxcpio`, `oxcat`, and `oxunzip`.
